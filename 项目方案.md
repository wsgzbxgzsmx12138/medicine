## 1. 项目背景与目标

### 1.1 背景

卡尤迪生物科技宜兴有限公司在向 NMPA（国家药品监督管理局）申报境内第三类体外诊断试剂注册时，需准备大量合规文件，包括：

- 监管信息（章节目录、申请表、产品列表、沟通记录、声明文件等）
- 综述资料、非临床资料、临床评价资料
- 产品说明书与标签样稿
- 质量管理体系文件

本次模拟以申报产品「**呼吸道合胞病毒、肺炎支原体核酸检测试剂盒（荧光PCR法）**（第三类 IVD，分类编码 6840-23-002）」为场景，数据来源为已有的「新型冠状病毒 2019-nCoV 核酸检测试剂盒（荧光PCR法）」说明书，目标是从源文件自动提取、填写并核查目标注册文件。

### 1.2 核心需求（来源：AI工具流程&需求.doc）

| 需求类型 | 说明 | 涉及章节 |
|----------|------|----------|
| **类型一** | 将文件夹中的文件自动汇总文件目录，并识别每份文件页数 | CH1.2 章节目录 |
| **类型二** | 从产品说明书提取信息，自动填写到申报文件中 | CH1.4 申请表、CH1.5 产品列表、CH1.11.1 标准清单 |
| **类型三** | 搜集与目标产品和检测靶标相关的行业指南文件，列入文件中 | CH1.11.1 标准清单（扩展） |

### 1.3 AI Agent 五大核心功能目标

```
① 自动汇总注册申报文件夹中的所有文件及页数
② 对照 NMPA 法规要求核查文件完整性并预警缺失
③ 提取产品关键信息并自动填写至申报文件
④ 核查文档结构与信息一致性
⑤ 输出合规风险预警与处理建议
```

## 3. 技术栈选型

### 3.1 设计原则：双轨实施，Demo 优先

采用 **「轻量 Demo 轨 + 可演进完整轨」**，避免一开始陷入平台化或大框架过重的问题：

| 轨道 | 定位 | 适用阶段 |
|------|------|----------|
| **Demo 轨（优先）** | Streamlit UI + `core/` 纯 Python 流水线 | 复试 POC、3～5 天可跑通 |
| **完整轨（演进）** | LangGraph 工作流 + 模块化 `src/` | 功能扩展、架构答辩、V1.0+ |

两条轨道 **共用同一套核心资产**：`config/` 规则库、`prompts/` 提示词、`data/` 样本与输出规范。Demo 轨不是临时 hack，而是完整轨的最小可验证子集。

### 3.2 技术栈总览

```
语言运行时:    Python 3.11+

【Demo 轨 — 复试展示优先】
交互界面:      Streamlit（上传文件夹、进度条、表格高亮、报告下载）
编排方式:      纯 Python 顺序调用 / 轻量 LangChain（不做 ReAct 自主循环）
数据展示:      Pandas DataFrame → Streamlit 表格 / Excel 导出

【完整轨 — 架构演进】
Agent 框架:    LangGraph（有向图工作流，条件分支、状态共享）
数据持久:      SQLite（任务状态，Demo 轨可省略）

【两轨共用】
AI/LLM:        DeepSeek-V3 / 智谱 GLM-4（中文医疗文本、成本低，Demo 首选）
               OpenAI GPT-4o（structured output 稳定，备选）
文档处理:      python-docx（.docx 读写）、PyMuPDF（PDF 文本与页数）
表格/比对:      Pandas（内存比对、目录汇总）、openpyxl（CH1.2 Excel 输出）
规则配置:      JSON（nmpa_rules.json、field_mapping.json）
配置管理:      python-dotenv
日志/测试:      loguru、pytest
```

### 3.3 技术选型理由

| 组件 | Demo 轨选择 | 完整轨演进 | 理由 |
|------|-------------|------------|------|
| **UI** | **Streamlit** | 可保留或换 FastAPI+前端 | 几十行即可出专业后台感，复试演示远胜 CLI；上传/高亮/下载一体化 |
| **编排** | **纯 Python pipeline** | LangGraph | Demo 要逻辑闭环，不要编排复杂度；完整轨再引入图结构便于扩展 |
| **LLM** | **DeepSeek / GLM-4** | 多模型可切换 | 中文医疗器械文本理解好、API 成本低；提取用 LLM，判定用规则 |
| **文档** | python-docx + PyMuPDF | 同左 | 与本题 10 份资料格式匹配 |
| **比对输出** | **Pandas** | Pandas + JSON 报告 | 目录表、一致性矩阵在 UI 中直接展示 |
| **规则库** | JSON 本地配置 | + 在线 CMDE 爬取 | Demo 不依赖外网；规则是核心资产，不是 LLM 临场发挥 |

### 3.4 关键架构分工（借鉴「LLM 提取 + 规则判定」）

```
LLM 负责：语义理解、非结构化段落抽取、报告措辞润色
规则引擎负责：完整性有无、字段字符串一致性、章节必检项、合规最终判定

医疗注册场景：合规结论不容幻觉 → 判定权必须在规则层，LLM 只做辅助输入。
```

## 6. 核心流程设计

### 6.1 Demo 五阶段流水线（`app.py` 主路径）

与参考方案对齐的命名，便于复试口述；比原七节点合并为五步，**自动填写**并入第四阶段：

| 阶段 | 模块 | 输入 | 输出 | UI 反馈 |
|------|------|------|------|---------|
| **一、Scanner** | `scanner.py` | `data/upload/` | 文件清单 + 页数 + CH1.2 Excel | Pandas 表格展示 |
| **二、Rule Engine** | `checker.py` | 清单 + `nmpa_rules.json` | 缺失文件列表 | 缺失项红色高亮 |
| **三、Extractor** | `extractor.py` | 说明书等 docx 文本 | `extracted_fields.json` | 字段对照表 + 置信度 |
| **四、Cross-Checker + Filler** | `checker.py` + `filler.py` | 多文档 JSON 字段 | 冲突项 + 已填写 docx | 冲突黄色/红色高亮 |
| **五、Reporter** | `reporter.py` | 缺失 + 冲突汇总 | `风险预警报告.md` | 可下载；LLM 仅润色措辞 |

```
upload/ → Scanner → Rule Engine → Extractor → Cross-Checker/Filler → Reporter → output/
```

### 6.2 完整轨工作流（LangGraph 有向图，V1.0 演进）

```
START
  │
  ▼
[N01: 文件扫描]
  • 遍历 data/input/ 目录
  • 统计文件名、格式、页数
  • 生成文件目录汇总（CH1.2 所需）
  │
  ▼
[N02: 完整性核查]
  • 对照 nmpa_rules.json（来自附件4）
  • 检查 6 大章节所需文件是否存在
  • 输出缺失文件列表 + 严重等级
  │
  ├──[有严重缺失]──→ 生成预警，继续执行
  │
  ▼
[N03: 信息提取]
  • 读取「目标产品说明书.docx」
  • LLM 结构化提取 ProductInfo 字段
  • 验证字段完整性，缓存到 extracted_fields.json
  │
  ▼
[N04: 自动填写]
  • 读取各目标模板文件（CH1.4, CH1.5, CH1.11.1）
  • 按 field_mapping.json 将提取字段写入模板
  • 输出已填写的 .docx 文件
  │
  ▼
[N05: 一致性核查]

### 6.2 异常处理策略

| 场景 | 处理策略 |
|------|----------|
| 文件无法读取 | 记录警告，跳过该文件，继续流程，在报告中标注 |
| LLM 提取字段缺失 | 尝试正则兜底提取；若仍缺失，在报告中标注「需人工确认」 |
| 网络请求失败 | 使用本地 standards_list.json 缓存数据，标注「可能非最新」 |

### 8.1 规则库设计策略：代表项 + 全量分层（不照搬「仅 3 条规则」）

### 8.2 Prompt 设计原则（核心资产，防幻觉）

提取 Prompt（`prompts/extraction_prompt.md`）必须包含：

1. **角色边界**：「你是医疗器械注册资料审核助手，只依据给定文本作答」  
2. **禁止编造**：找不到则返回 `"未提及"`，不得推断  
3. **结构化输出**：严格 JSON Schema，便于规则层消费  
4. **原文锚点**：要求标注来源章节（如 `【预期用途】`）  
5. **先规则后 LLM**：说明书 `【产品名称】` 等固定格式优先正则提取，LLM 只处理剩余字段  

报告润色 Prompt（可选）：仅将规则引擎已确认的缺失/冲突改写成可读建议，**不得新增风险项**。

示例约束句：

> 如果文本中找不到对应信息，请返回 `"未提及"`，绝不能编造。提取目标：预期用途、储存条件、包装规格、检测靶标……

- 展示 Scanner 输出：10 个文件、页数、CH1.2 汇总表  
- 强调：Pandas + PyMuPDF + python-docx，无手工统计  

**2. 完整性初筛（1 分钟）**

- 展示 Rule Engine：CH1.6 营业执照缺失 → 红色行  
- 说明：规则来自附件4 结构化，不是 LLM 猜的  

**3. 信息提取与自动填写（2 分钟）**

- 展示 Extractor：说明书 → JSON 字段对照表  
- 打开 `CH1.4_申请表_已填写.docx`，对比填写前后（**体现需求类型二**）  

**4. 故意制造冲突（1 分钟，高分项）**

- 预备两套测试数据：`正常集` / `冲突集`（如 CH1.4 与说明书产品名称故意不一致）  
- 运行 Cross-Checker → UI 显示冲突 + 报告原文引用  
- 口述：**合规判定走规则引擎，LLM 不参与是否一致的最终结论**  

**5. 技术复盘（1 分钟）**

- 为什么用 LLM：章节格式乱、表述不统一，正则覆盖不全  
- 为什么用规则：医疗注册不能幻觉，缺失/冲突必须可审计  
- 架构演进：Demo 用 `core/` 流水线；完整轨可升级为 LangGraph 编排，模块职责一一对应

### V1.0（下一个优先迭代目标）

**需要投入的技术资源：**
- CMDE 官网爬虫开发（约3天）
- 指南文件解析与结构化（约2天）
- 增量更新机制 + 版本管理（约2天）
- 总计：约1周开发 + 测试

## 13. 风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| LLM 字段提取出现幻觉 | 中 | 高 | 规则提取优先，LLM 兜底；提取置信度标注；人工复核提示 |
| NMPA 官网结构变化导致爬虫失效 | 低 | 中 | 本地规则库作为备份；爬虫设置告警 |
| .doc 格式无法解析 | 高 | 低 | 要求输入文件统一转 .docx；提供转换脚本 |
| 模板结构变更导致填写定位失败 | 中 | 高 | 使用书签/内容控件定位（非坐标定位）；定位失败时输出警告而非崩溃 |
| 法规更新导致规则过期 | 中 | 高 | 规则库版本管理；定期更新；V1.0 实现在线对接 |
| 大文件导致 LLM 上下文超限 | 低 | 中 | 分块处理；优先使用规则提取；仅对困难段落调用 LLM |

## 14. 从零搭建可行性评估

### 14.1 项目定位

本项目为**完全独立**的 NMPA 注册资料准备与审核 Agent，不依赖任何企业内部平台或既有 Agent 中台。交付形态为：

- **Demo 轨**：Streamlit UI + `core/` 五阶段 Python 流水线  
- **完整轨（可选演进）**：LangGraph 工作流 + 模块化 `src/`  

核心资产是 `config/` 规则库、`prompts/` 提示词与 `core/` 业务逻辑，而非外部平台集成。

### 14.2 从零搭建是否现实？

**结论：现实，且符合题目预期。**

题目要求「设计并实现（或详细描述）一个智能体」并配合复试 Demo，评估重点在于：

1. 能否把业务拆成可执行工作流（扫描 → 校验 → 提取 → 填写 → 核查 → 预警）  
2. 能否选对工具（python-docx、规则引擎、LLM structured output、Streamlit）  
3. 能否用真实 10 份资料跑出可展示结果  
4. 能否说清 Demo 与完整产品的边界  

**不需要**自建通用 Agent 平台。Streamlit + `core/` 流水线即可覆盖 Demo；LangGraph 仅作为完整轨的可选演进路径。

### 14.3 工作量拆分（聚焦必做项）

```
本项目核心交付                    Demo 阶段暂不建设
─────────────────                ─────────────────
① 五阶段业务工作流                多租户 / 权限 / 登录
② docx 读写 + 页数统计            通用 Agent 编排平台
③ NMPA 规则库 (JSON)              可视化工作流设计器
④ 字段映射 + 模板填写             长期记忆 / 多轮对话
⑤ LLM 提取 + 一致性校验           企业级监控告警
⑥ Streamlit 展示 + 报告生成        与外部审评系统对接
```

### 14.4 推荐的最小可行 Demo 形态

```bash
streamlit run app.py
# 上传 data/upload/normal → 目录表 + 已填写 docx + 风险报告.md
```

约 **3～5 天**可做出 Streamlit 可演示闭环；**1～2 周**可补齐 filler、三套测试集与答辩材料。

### 14.5 复试 Narrative 建议

> 「这是一个独立的 NMPA 注册资料审核 Agent POC：Streamlit 负责交互展示，`core/` 五阶段流水线负责业务逻辑，`config/` 与 `prompts/` 是可审计的核心资产。Demo 证明垂直 Agent 闭环；LangGraph 是完整轨演进选项，不是复试必选项。」

## 15. 参考方案评价与融合策略

> 对「轻量脚本 + Streamlit 可视化」外部方案的取舍说明，便于答辩时解释设计决策。

### 15.1 值得借鉴的精华

| 要点 | 融合方式 |
|------|----------|
| **放弃平台化，先做 POC 闭环** | 采纳；§3.1 双轨策略，Demo 轨优先 |
| **Streamlit 作复试 UI** | 采纳；`app.py` 为主入口，比 CLI 更适合演示 |
| **core/ 模块化** | 采纳；scanner / checker / extractor 等职责单一，见 §4.3 |
| **五阶段命名清晰** | 采纳；Scanner → Rule Engine → Extractor → Cross-Checker → Reporter |
| **Pandas 表格展示与 Excel 导出** | 采纳；CH1.2 目录与 UI 表格 |
| **规则 JSON + Prompt 即核心资产** | 采纳并加强；增加分层规则与防幻觉 Prompt |
| **复试剧本：亮边界 → 跑流程 → 造冲突 → 技术复盘** | 采纳；见 §11.2 |
| **DeepSeek / GLM 等国产模型** | 采纳为 Demo 默认；保留 GPT-4o 备选 |
| **LLM 抽取 + 规则判定分离** | 采纳；§3.4 明确判定权在规则层 |

### 15.2 不采纳或需修正的糟粕

| 参考方案表述 | 问题 | 本方案处理 |
|--------------|------|------------|
| 「仅 3 条 nmpa 规则」 | 过浅，无法体现 NMPA / 附件4 专业度 | 分层：Demo 代表项 + CH1 全量 + 附件4 扩展 |
| 未提 **自动填写 CH1.4/CH1.5** | 遗漏本题需求类型二 | 保留 `filler.py`，Demo 必演示 |
| 未提 **CH1.2 正式目录格式** | 遗漏需求类型一 | scanner 输出对齐 RPS 目录表 |
| 「LangChain 或纯 Python 即可，不必上框架」 | 部分正确但过于绝对 | Demo 纯 Python；完整轨保留 LangGraph 演进叙事 |
| 一致性「字符串绝对一致」 | 方向对，但缺少 normalize | 保留并增加 normalize（空格、全半角） |
| LLM 润色报告若替代判定 | 合规风险 | LLM 只润色，不新增风险项 |
| 「几十行 Streamlit 就够」 | 易低估 filler 与 docx 表格难度 | 工期仍按 3～5 天 POC / 1～2 周稳态估算 |
| 未覆盖需求类型三 | 缺标准清单扩展 | standards_list.json + V1.0 CMDE |

### 15.3 融合后的定位一句话

**Streamlit 负责「看得见的 POC」，core/ + config/ + prompts/ 负责「可审计的业务内核」，LangGraph 负责「答辩后的架构演进叙事」。**
